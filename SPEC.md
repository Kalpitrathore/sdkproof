# SDKProof — v1 Spec

_Last updated: 2026-07-16_

## What it is
A CLI harness that measures how well today's top coding models use a library's **real, current** API, and reports a 0–100 score plus the exact API calls they get wrong. The public "AI-readiness scorecards" it produces are both the product and the distribution engine.

## v1 goal (this build)
**Prove the signal on one SDK (Prisma).** Produce one credible Prisma scorecard we can eyeball to answer a single go/no-go question:

> Are the failures the harness surfaces genuinely real (a Prisma maintainer would nod) and interesting enough to justify publishing 5 scorecards + a Show HN?

Everything is scoped to answering that question cheaply, not to shipping a business.

## Decisions (locked)
- **Ecosystem:** TypeScript / npm first. `tsc` gives a precise, cheap correctness signal against real installed types.
- **Code under test is generated by model APIs:** Claude + GPT (current flagship coding models), called directly. Not agent CLIs (deferred).
- **Verification is type-check-only.** We measure "does the code use the real, current API surface correctly," NOT "does it run." Stated honestly on every scorecard.
- **Curated tasks.** taskgen proposes ~30; a human keeps the best ~15–20. Task quality = scorecard credibility.
- **Reproducible by construction.** Pin library version + model versions; low temperature; cache tasks, candidates, and verdicts as files.

## Non-goals (explicitly deferred)
Interactive web tool · runtime/execution checks · auth-gated SDKs · real agent-CLI orchestration · the README badge · more than 2 models · hosted monitoring/billing SaaS · the polished public scorecard site. None of this until Prisma proves the signal.

## Architecture — 6 small, independently-testable units

```
library spec ─▶ [1 taskgen] ─▶ tasks.json ─▶ [2 generate] ─▶ candidates.json
                                                                    │
 scorecard.md ◀─ [6 report] ◀─ result.json ◀─ [5 score] ◀─ [4 classify] ◀─ [3 verify]
```

1. **taskgen** — `(LibrarySpec) → Task[]`. LLM turns Prisma docs into ~30 realistic, self-contained tasks; output cached to `tasks.json` for human curation. `Task = { id, area, difficulty, prompt, skeleton }`.
2. **generate** — `(Task, Model) → Candidate`. Calls a model adapter for a TypeScript solution. `Candidate = { taskId, model, code }`. Temp ≈ 0.
3. **verify** — `(Candidate) → Verdict`. Writes the candidate into the sandbox fixture, runs `tsc --noEmit`, captures diagnostics. `Verdict = { taskId, model, passed, errors: TscError[] }`. **Load-bearing unit — gets the most test coverage.**
4. **classify** — `(TscError[]) → FailurePattern[]`. Buckets errors: hallucinated member (TS2339), wrong arg count (TS2554), bad import (TS2307), type mismatch (TS2322/2345), deprecated API. Rule-based on error codes + LLM for the human label + representative snippet.
5. **score** — `(Verdict[]) → Result`. Per-model pass rate, overall library score (0–100), ranked top failure patterns. → `result.json`.
6. **report** — `(Result) → scorecard.md`. Human-readable scorecard for eyeballing. (Polished public page deferred.)

## The Prisma fixture (the one tricky bit)
Prisma needs a schema + codegen before its types exist. So `fixtures/prisma/` is a real project:
- `schema.prisma` with a representative model (User, Post, Profile; 1-1, 1-many, a compound unique).
- Pinned `@prisma/client` version; client generated once during `setup`.
- Strict `tsconfig.json`.
- Each candidate is written to `candidate.ts` and type-checked against this.

Prisma is deliberately the **hardest** fixture (codegen). Later libs (Drizzle, Clerk, Better-Auth) are plain installs — easier.

## Scoring definition
- **Pass** = the candidate compiles clean under `tsc --noEmit` in the fixture (zero errors).
- **Fail** = any diagnostic; errors are classified in unit 4.
- **Anti-gaming:** the task skeleton forces real Prisma usage (e.g. "return the created record"); a stub/empty solution can't pass. verify additionally asserts the candidate references the library import.
- **Score** = round(100 × passing_tasks / total_tasks), reported overall and per-model, with the ranked failure patterns.

## Tech stack & layout
Node 20 + TypeScript, `tsx` runner, `@anthropic-ai/sdk` + `openai` for generation, the `typescript` compiler API for verify, plain JSON/markdown storage (no DB).

```
sdkproof/
  SPEC.md
  package.json
  tsconfig.json
  src/
    types.ts            # shared types (Task, Candidate, Verdict, Result, ...)
    libraries/prisma.ts # LibrarySpec for Prisma
    taskgen.ts  generate.ts  verify.ts  classify.ts  score.ts  report.ts
    models/anthropic.ts  models/openai.ts  models/index.ts
    cli.ts              # `sdkproof run --lib prisma`
  fixtures/prisma/      # schema.prisma, tsconfig, generated client
  data/                 # tasks.json, candidates.json, result.json (cached)
  scorecards/           # prisma.md
  test/                 # unit tests (verify/classify/score first)
```

## Milestone-1 acceptance (go/no-go)
- `sdkproof run --lib prisma` runs end-to-end from cached tasks and emits `scorecards/prisma.md`.
- The scorecard shows a per-model score + top failure patterns with real offending snippets.
- **Human judgment:** eyeball 10 failures — are they real API mistakes (not harness bugs)? If yes → scale to 5 SDKs + publish. If the failures are mostly harness noise → fix verify, or reconsider.

## Risks / open questions
- **False fails from harness strictness** (unrelated tsc errors, not library API). Mitigation: minimal, controlled fixture + tsconfig; classify separates library-API errors from environment errors; manual audit in milestone 1.
- **Task realism.** Mitigation: human curation step is mandatory, not optional.
- **Model IDs / params.** Confirm current Claude + GPT coding-model IDs from the Claude API reference before writing `models/*`.
